Single-process entry `src/index.ts` bootstraps a Hono app with global middleware: CORS headers, then an `/api/*` guard that delegates to `requireAuth`, which decodes a self-implemented HS256 JWT (`src/auth.ts`) and attaches `userId`/`email` via Hono's typed `Variables`. Route modules are co-located per domain — `auth.ts` (JWT sign/verify, password hashing, user CRUD), `keys.ts` (API key lifecycle), `providers.ts` (BYOK provider registry), `conversations.ts` (conversation/message store), `usage.ts` (analytics & cost recording) — all sharing one shared `db.ts` singleton that opens `bun:sqlite` in WAL mode and runs DDL on import. The DB schema defines six tables (`users`, `api_keys`, `providers`, `usage`, `conversations`, `messages`) with foreign keys enforced at the engine level. A local `fakeModelReply` plus SSE `/api/stream` endpoint provide a streaming chat demo without external LLM calls. All JSON payloads are validated through `@hono/zod-validator` against inline Zod schemas; responses are plain JSON objects. The process is served via `Bun.serve({ fetch: app.fetch })` and configurable through `PORT`, `CORS_ORIGIN`, `BACKEND_DB`, and `JWT_SECRET` env vars.
