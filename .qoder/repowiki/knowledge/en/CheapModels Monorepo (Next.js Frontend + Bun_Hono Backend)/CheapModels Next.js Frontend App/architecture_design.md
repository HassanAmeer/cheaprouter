The app is split by concern across four sibling directories under `src/`:
- `app/` — Next.js App Router routes. `layout.tsx` is the single root layout that wraps every page with `ThemeProvider`, `ToastProvider`, and `AuthProvider`, so theme, toast notifications, and user session are globally available. Routes are partitioned into public marketing pages (`login`, `signup`, `pricing`, `docs`, `privacy`, `terms`), protected dashboard routes (`dashboard/*`) guarded by the auth provider, and interactive demos (`chat`, `cli`). Server-side data is accessed exclusively through `lib/api.ts`; no route or component issues raw fetches.
- `components/` — shared React pieces consumed by both public pages and dashboard views. Marketing shells (`SiteNav`, `LegalShell`, `HeroTerminal`, `BranchFeatures`, `InstallBox`) sit alongside reusable `ui/` primitives (`SpaceButton`, `Toast`, `Primitives`, charts, pixelated canvas) and feature components like `ModelsTable`.
- `config/theme.ts` — design tokens (colors, fonts, spacing) applied at the root `<html>` via the Inter font variable; consumed by CSS Modules throughout.
- `lib/` — cross-cutting runtime: `api.ts` (typed request wrapper reading `NEXT_PUBLIC_API_URL` and injecting a Bearer token from `localStorage('cm_token')` plus an SSE-based `streamChat` helper), `db.ts` (SQLite client used by server routes), and `utils.ts`.

Cross-child wiring: pages in `app/` import components from `components/`, read theme tokens from `config/theme.ts`, and call methods on `lib/api.ts`; server routes under `app/api/` use `lib/db.ts` to persist state. There is no separate service layer — the boundary between client and server is the `/api/*` route tree, which the client never calls directly but goes through the typed wrapper instead.
